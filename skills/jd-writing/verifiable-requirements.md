# 把用人标准写成可验证要求

任职要求不能停在"熟悉 X、了解 Y"。要写成能被 [[resume-screening]] 和 [[interview-evaluation]] 验证的要求。

## 能力 → JD 表达 → 面试验证

| 能力 | JD 表达 | 面试验证 |
| --- | --- | --- |
| 主动性 | 能主动定义问题并推动方案，而不是被动等待详细任务 | 追问过去主动发现并推动问题的案例 |
| 问题拆解 | 能把开放问题拆成目标、约束、路径和指标 | 给一个复杂场景现场拆解 |
| 结果意识 | 关注用户价值、验收标准和业务 / 产品闭环 | 追问项目如何验收、指标是什么 |
| 评估意识 | 能设计实验、指标、对照、回测或复盘 | 追问如何证明方案有效 |
| 工程 / 交付意识 | 能从 Demo、方案或想法走向稳定可用的交付结果 | 追问日志、监控、回滚、服务稳定性或交付验收 |
| 协作沟通 | 能跨产品、研发、销售、客户、运营等角色推进闭环 | 追问冲突协调和跨团队推进案例 |
| 学习能力 | 能快速进入新行业、新业务或新技术栈 | 追问最近一次快速学习并落地的案例 |
| AI Native | 使用 AI 工具做调研、编码、调试、分析、写作或内容生产 | 要求讲一个完整 AI 协作案例 |

## 使用规则

- 不同岗位类型选择不同能力重点，**不要把所有能力机械塞进每一份 JD**。
- 每条任职要求至少能匹配一种简历信号 + 一种面试问题。
- 简历层面无法验证的要求（如"心态稳"、"踏实"）应放进面试重点而不是任职要求。

## 反例

| 模糊表达 | 问题 | 修正 |
| --- | --- | --- |
| 熟悉大模型 | 不可验证；几乎所有候选人都会写"熟悉" | "有大模型应用项目交付经验，能说清数据、Prompt、评测、上线和监控" |
| 了解 Agent 系统 | "了解"等于没说 | "做过 Agent 系统的任务定义、状态管理、执行回放、自动评测或回归测试中的至少一项" |
| 有责任心 | 无法在简历或面试观察 | "对线上稳定性 / 用户反馈 / 业务指标有主动追踪和复盘的习惯" |
| 抗压能力强 | 主观、易被表演 | "有从 0 到 1、紧急上线或多线并行项目的交付经验" |
| 沟通能力强 | 几乎所有候选人都会写 | "能独立面向客户 / 用户 / 跨部门做演示、需求澄清或结论汇报" |
