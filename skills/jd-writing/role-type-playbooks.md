# 岗位类型写作 Playbook

根据岗位类型突出重点。**不要**把所有能力机械塞进每一份 JD。

## 技术研发类

重点：工程能力、系统设计、交付稳定性、问题拆解。

应体现：

- 负责什么系统、模块或能力。
- 面对什么规模、复杂度、性能、稳定性或质量挑战。
- 使用哪些核心技术栈，但要说明技术解决什么问题。
- 如何交付、评估、监控和迭代。

避免：把岗位写成单纯技术名词堆叠。

## 算法 / 数据 / AI 类

重点：建模能力、数据理解、评估意识、服务化落地。

应体现：

- 数据来源、问题定义和目标变量。
- 算法、模型、特征、评估和实验设计。
- 离线效果如何转化为在线服务或业务价值。
- 模型监控、误差分析、可解释性和迭代机制。

避免：只写"准确率""模型训练"而不写评估、上线和业务闭环。

## 产品经理类

重点：需求洞察、产品定义、用户价值、优先级判断、跨团队推进。

应体现：

- 面向什么用户 / 客户 / 业务场景。
- 负责从问题发现、需求定义、方案设计到上线复盘的闭环。
- 如何做数据分析、竞品分析、用户调研和效果评估。
- 如何与研发、设计、运营、销售或客户协作。

避免：只写"写 PRD、跟进项目"这类低辨识度职责。

## 产品市场 / 运营 / 增长类

重点：把产品能力转化为用户价值、市场表达和增长结果。

应体现：

- 产品定位、核心卖点、竞品对比。
- 典型场景、解决方案、客户案例。
- 官网、产品手册、PPT、白皮书、演示脚本、内容资产。
- 活动、内容、渠道、用户反馈和转化闭环。
- AI 工具驱动的内容生产、竞品分析和市场洞察。

避免：让岗位看起来只是活动执行或普通文案岗。

## 解决方案 / 售前 / 交付类

重点：行业理解、客户沟通、方案设计、项目落地。

应体现：

- 面向哪些客户或行业场景。
- 如何识别客户痛点并转化为解决方案。
- 如何组织演示、方案、标书、POC 和交付验收。
- 如何把客户反馈回流给产品和研发。

避免：只写"配合销售、写方案"，要体现独立判断和闭环能力。

## 研究 / 战略 / 分析类

重点：信息搜集、结构化分析、判断力、输出质量。

应体现：

- 关注的行业、政策、市场、技术或竞争格局。
- 如何构建分析框架、验证信息和形成判断。
- 输出什么报告、策略、洞察或决策建议。
- 如何支撑产品、销售、投资、战略或管理决策。

避免：只写"收集资料、撰写报告"，要体现独立分析和判断。
