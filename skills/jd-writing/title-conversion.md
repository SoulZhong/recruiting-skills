# 内部岗位名 → 行业通用岗位名

岗位名称要使用候选人在招聘平台会搜索、能快速理解的名称。**不要**把内部项目名、模块名、组织黑话或小众英文概念放进主标题。

## 常见转换示例

| 内部说法 | 候选人外发岗位名示例 |
| --- | --- |
| 某内部系统 / 平台模块开发 | 后端开发工程师 / 平台开发工程师 / 全栈工程师 |
| 某算法能力服务开发 | 算法工程师 / 机器学习工程师 / AI 算法工程师 |
| 某 Agent / LLM 应用模块 | AI Agent 工程师 / 大模型应用开发工程师 / LLM 应用工程师 |
| 某数据链路或信号模块 | 数据开发工程师 / 数据平台工程师 / 数据工程师 |
| 某业务增长和内容包装 | 产品运营经理 / 产品市场经理 / 增长运营经理 |
| 某售前材料和客户方案 | 解决方案经理 / 售前解决方案顾问 / 产品市场经理 |
| 某研究型岗位 | 研究员 / 行业研究员 / 算法研究员 / 策略研究员 |
| 某项目统筹岗位 | 项目经理 / 产品经理 / 交付经理 |

## 转换原则

1. **以候选人搜索词为准**：在主流招聘平台上能被检索到的名称优先。
2. **避免自造词**：有行业通用名时不要发明新岗位名。
3. **新概念下沉到职责**：Agent、Harness、Skill、LoRA 等小众或新概念可以出现在岗位职责、加分项中，但必须用通俗语言解释。
4. **避免英文 + 缩写堆叠**：候选人对内部缩写没有上下文，主标题保持中文 / 通用英文。
5. **岗位层级标在岗位名后**：如 `算法工程师（资深）`、`产品经理（校招）`，便于平台分流。

## 反例

把下表里的 "<内部代号>" / "<内部项目>" / "<内部产品>" 当成你自己组织里实际的内部叫法（如平台代号、模块名、产品代号、内部体系名）替换思考。

| 反例 | 问题 | 修正 |
| --- | --- | --- |
| `<内部代号>` 工程师 | 内部术语，候选人不知道是做什么 | AI Agent 工程师 / Agent 系统平台工程师 |
| `<内部项目>` 产品经理 | 内部项目代号 | 大模型应用产品经理 / AI 决策产品经理 |
| 多智能体协同方向研究员 | 看起来像论文方向，不像岗位 | 多智能体系统研究员 / 多智能体算法研究员 |
| `<内部产品>` 体系负责人 | 内部产品概念 | AI Workflow 平台产品负责人 |
| AIGC 提效平台开发 | 内容模糊 | AI 应用开发工程师 / AIGC 平台工程师 |
